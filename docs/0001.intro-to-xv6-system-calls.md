# Intro to xv6 System Calls 

## What is a System Call in an Operating System?

A system call is a mechanism that allows a user-space program to request a service from the operating system's kernel. It acts as an interface between the application and the operating system, enabling programs to interact with hardware or perform privileged operations that are restricted to the kernel.

## Key Characteristics of System Calls:
1. **Kernel Mode Execution**: System calls transition the CPU from user mode to kernel mode, allowing the program to execute privileged instructions.
2. **Predefined Interface**: Operating systems provide a set of predefined system calls for common operations like file manipulation, process control, and inter-process communication.
3. **Hardware Abstraction**: System calls abstract hardware details, providing a consistent interface for applications.

## Common Categories of System Calls:
1. **Process Control**: `fork()`, `exec()`, `exit()`
2. **File Management**: `open()`, `read()`, `write()`, `close()`
3. **Device Management**: `ioctl()`, `read()`, `write()`
4. **Information Maintenance**: `getpid()`, `alarm()`, `sleep()`
5. **Communication**: `pipe()`, `shmget()`, `mmap()`

## Example Workflow:
1. A user-space program invokes a system call (e.g., `read()`).
2. The system call triggers a software interrupt or trap, switching the CPU to kernel mode.
3. The kernel performs the requested operation (e.g., reading data from a file).
4. The kernel returns the result to the user-space program and switches back to user mode.

System calls are a fundamental concept in operating systems, enabling secure and controlled interaction between user applications and the system's hardware resources.

## xv6-Specific Implementation Details

### Program Termination in xv6
All xv6 programs don't just return from main - they actually call `exit()`. If you don't put an exit call at the end of your main program, something slightly weird will happen (to be discussed later in the semester).

### The `exit()` System Call
- **Purpose**: Tells the kernel "I'm done"
- **Implementation**: Looks like a function call, but eventually becomes a trap instruction
- **Note**: In xv6, the trap instruction is called `int` (for interrupt), which can be confusing

### How to Compile New User-Space Calls
1. **user.h**: Contains user-space call declarations
2. **Makefile**: Add your program to the `UPROGS` list with the format `_{function_name}`

## RISC-V Calling Conventions and Register Usage

### Register Usage for System Calls
For each system call, we move a number into register `a7`.

**a7 Register Background:**
- One of the argument registers in RISC-V
- Used to pass the system call number
- **Primary use**: System call identification

**Return Code Convention:**
- When you call a C function that returns an `int`, the return value is passed in `a0`
- Example: `int function_name()` - return value goes in `a0`
- **Important**: Arguments are passed in registers `a0`, `a1`, `a2`, etc.

### System Call Number Convention
- **Purpose**: Identify which system call to execute
- **Method**: Put the system call number in `a7` register
- **xv6 Convention**: Uses specific numbers for each system call
- **Routing**: The kernel uses this number to route to the correct trap handler

---

## Calling Conventions and Parameter Passing

### xv6's Approach to Arguments
Every operating system has its own convention for how to tell it which system call you're executing. In xv6-riscv, the convention is:

1. **a7**: System call number
2. **a0, a1, a2, ...**: Arguments (passed in argument registers)

**Example with `open()` system call:**
- The kernel needs to know which file to open
- Arguments are passed in registers (not on stack like regular function calls)
- This follows the standard RISC-V calling convention

## The ECALL Instruction

### What is the ECALL Instruction?
The `ecall` instruction is a powerful instruction that:
- Triggers a system call trap
- No arguments needed - system call number is in `a7` register
- Always routes to the kernel trap handler

### ECALL Instruction Execution Flow
When the `ecall` instruction is executed:

1. **State Preservation**: Processor state is saved in trap frame
2. **Mode Switch**: CPU switches from user mode to supervisor mode
3. **Handler Execution**: Starts executing at the kernel trap handler
4. **Routing**: The kernel uses the number in `a7` to route to correct system call
5. **Work Execution**: Performs the requested operation (e.g., opens a file)
6. **Return**: Returns to user space via `sret` instruction

### Return Value Convention
- **Return Register**: Return value is placed in `a0` register
- **System Call Return**: Whatever system call was executed returns its result in `a0`
- **Function Return**: The normal RISC-V return instruction uses `a0` for return values

## Practical Implementation: Adding System Calls

### The Three-Instruction Pattern
Adding a system call involves three main instructions:
1. **Move**: Put user-space call number into `a7`
2. **Trap**: Issue the `ecall` instruction
3. **Return**: Return from the function


### Header File Declarations (user.h)
The `user.h` file is crucial for system call implementation:

**Purpose:**
- Describes the signature of each system call
- Specifies what they return and what parameters they take
- Can include `void` for no parameters (like `exit()` in xv6)




---

## Summary

The system call mechanism in xv6 involves:
1. **Register Setup**: System call number in `a7`, arguments in `a0`, `a1`, etc.
2. **ECALL Execution**: Instruction that switches to supervisor mode
3. **Kernel Processing**: Routing to correct handler and performing requested operation
4. **Return**: Result placed in `a0`, return to user space

This three-instruction pattern (move, ecall, return) is the foundation of all system calls in xv6, making it a simple yet powerful mechanism for user-kernel communication.

# Kernel-Side Implementation of System Calls

## Kernel Boot Process and Initialization

### Overview
The kernel boot process is more complex than user-space programs. While you could start at the boot code to understand how everything initializes, the boot process follows a very prescribed set of actions. The computer follows specific steps to initialize the machine.

### Boot Sequence
1. **boot/main.c**: Eventually calls into the main routine (`main.c`)
2. **Single Processor Initialization**: This code runs on only one processor in a multiprocessor system
3. **Additional Processor Initialization**: Some initialization is done on other processors if multiple processors are available

### Key Initialization Tasks
The kernel initialization code performs several critical setup tasks:

```c
// Example initialization tasks in main.c
    kinit();         // physical page allocator
    kvminit();       // create kernel page table
    kvminithart();   // turn on paging
    procinit();      // process table
    trapinit();      // trap vectors
    trapinithart();  // install kernel trap vector
    plicinit();      // set up interrupt controller
    plicinithart();  // ask PLIC for device interrupts
    binit();         // buffer cache
    iinit();         // inode table
    fileinit();      // file table
    virtio_disk_init(); // emulated hard disk
    userinit();      // first user process
```

### Generic Kernel Protocol
The kernel follows a generic protocol:
1. **Boot Phase**: Set up a bunch of infrastructure when the system boots
2. **Runtime Phase**: Use that infrastructure to prevent user programs from doing bad things

This initialization phase is where all the system call infrastructure gets established.

## Trap Vector Setup and Initialization

### The Role of Trap Vectors
To handle system calls, the kernel must tell the CPU where the trap vectors are located. Each CPU must be informed about the location of these trap handlers.

### trapinithart() Function
The `trapinithart()` function is responsible for setting up the trap vectors:

```c
// Located in trap.c
void trapinithart(void) {
    // Set the trap vector to point to kernelvec
    w_stvec((uint64)kernelvec);
}
```

### RISC-V Trap Vector Setup
RISC-V has a simpler approach to trap vectors:

```c
// Set supervisor trap vector
w_stvec((uint64)kernelvec);
```

**Key Points about RISC-V traps:**
- Uses a single trap vector register (`stvec`)
- Much simpler than x86's complex interrupt descriptor table
- All traps (including system calls) go through the same entry point
- The `kernelvec` function handles the initial trap processing

### Trap Vector Register
RISC-V uses control and status registers (CSRs) for trap handling:
- `stvec`: Supervisor trap vector base address
- `scause`: Supervisor cause register (identifies trap type)
- `stval`: Supervisor trap value (additional trap information)
- `sepc`: Supervisor exception program counter (return address)

### Hardware Requirements
The RISC-V hardware requires:
- Setting the `stvec` register to point to the kernel trap handler
- Proper privilege mode setup (supervisor vs user)
- Correct CSR access permissions

## Assembly Code Handling: kernelvec.s and trampoline.s

### The kernelvec.s File
The `kernelvec.s` file contains the kernel trap handler:

**Structure:**
- Single entry point for all kernel traps
- Handles interrupts while in kernel mode
- Saves minimal state and calls kerneltrap()

**Kernel Trap Handler:**
```assembly
kernelvec:
    # Save registers
    addi sp, sp, -256
    sd ra, 0(sp)
    sd sp, 8(sp)
    # ... save other registers ...
    call kerneltrap
    # Restore registers and return
```

### The trampoline.s File: User Trap Handler
User space traps are handled through the trampoline:

**Primary Purpose:**
- Handle traps from user space
- Switch between user and kernel page tables
- Save/restore user register state

**Critical Operations:**
1. **Page Table Switch**: Change from user to kernel page table
2. **Register Preservation**: Save all user registers to trap frame
3. **Stack Switch**: Switch from user stack to kernel stack
4. **Call into C Code**: Eventually calls `usertrap()` function

### Stack Switching During Traps
When a trap occurs, an important transition happens:

**User Stack vs Kernel Stack:**
- **User Stack**: Used by the process during normal execution
- **Kernel Stack**: Per-process stack used when running in kernel mode
- **Security Reason**: Kernel doesn't trust user stack for critical operations

**The Transition Process:**
1. Hardware automatically switches from user stack to kernel stack
2. Hardware saves some state (like program counter) on the kernel stack
3. `alltraps` saves additional registers on the kernel stack
4. All kernel operations use the kernel stack
5. On return, the process reverses to restore user stack

### RISC-V Trap Handling
RISC-V uses CSRs to communicate trap information:

**Trap Information:**
- `scause` register contains the trap cause (interrupt vs exception)
- `stval` register contains additional trap-specific information
- `sepc` register contains the program counter to return to

**Why Use CSRs?**
- Hardware automatically updates these registers on trap
- Kernel reads `scause` to determine trap type
- System calls are identified by specific cause codes

## Trap Frame Structure and State Saving

### What is a Trap Frame?
A trap frame is a data structure that contains all the processor state that needs to be saved when transitioning from user space to kernel space. It ensures that when the kernel returns control to the user program, everything is exactly as it was before the trap.

### The trapframe Structure
The trap frame structure is defined in `kernel/proc.h`:

```c
struct trapframe {
    /*   0 */ uint64 kernel_satp;   // kernel page table
    /*   8 */ uint64 kernel_sp;     // top of process's kernel stack
    /*  16 */ uint64 kernel_trap;   // usertrap()
    /*  24 */ uint64 epc;           // saved user program counter
    /*  32 */ uint64 kernel_hartid; // saved kernel tp
    /*  40 */ uint64 ra;
    /*  48 */ uint64 sp;
    /*  56 */ uint64 gp;
    /*  64 */ uint64 tp;
    /*  72 */ uint64 t0;
    /*  80 */ uint64 t1;
    /*  88 */ uint64 t2;
    /*  96 */ uint64 s0;
    /* 104 */ uint64 s1;
    /* 112 */ uint64 a0;
    /* 120 */ uint64 a1;
    /* 128 */ uint64 a2;
    /* 136 */ uint64 a3;
    /* 144 */ uint64 a4;
    /* 152 */ uint64 a5;
    /* 160 */ uint64 a6;
    /* 168 */ uint64 a7;
    /* 176 */ uint64 s2;
    /* 184 */ uint64 s3;
    /* 192 */ uint64 s4;
    /* 200 */ uint64 s5;
    /* 208 */ uint64 s6;
    /* 216 */ uint64 s7;
    /* 224 */ uint64 s8;
    /* 232 */ uint64 s9;
    /* 240 */ uint64 s10;
    /* 248 */ uint64 s11;
    /* 256 */ uint64 t3;
    /* 264 */ uint64 t4;
    /* 272 */ uint64 t5;
    /* 280 */ uint64 t6;
};
```

### Hardware vs Software State Saving
In RISC-V, the trap frame is built differently:

**Hardware-Saved State (Automatic):**
- Program counter saved in `sepc` CSR
- Cause of trap saved in `scause` CSR
- Additional trap value in `stval` CSR
- Previous privilege mode in `sstatus` CSR

**Software-Saved State (By kernel trap handler):**
- All 32 general-purpose registers (`x0` through `x31`)
- Kernel-specific fields (satp, sp, trap handler address)
- User program counter (`epc`)
- Kernel context information

### Why Save All This State?
- **Register Preservation**: User programs expect registers unchanged after system calls
- **Return Address**: Program counter tells us where to resume execution
- **Stack Information**: Need to restore user stack after kernel operations

### Per-Process Kernel Stacks
Each process has two stacks:
- **User Stack**: Normal program execution, function calls, local variables
- **Kernel Stack**: Kernel mode execution, trap frames, kernel function calls

**Security**: Kernel doesn't trust user stack - uses separate protected kernel stack

## C Code Handling: trap.c and syscall.c

### The trap() Function in trap.c
After all the assembly code completes, we finally reach C code! The `trap()` function in `trap.c` is the generic entry point for all traps:

```c
void trap(struct trapframe *tf) {
    if(tf->trapno == T_SYSCALL) {
        // Handle system call
        if(proc->killed)
            exit();
        
        proc->tf = tf;  // Save trap frame in process structure
        syscall();      // Call the system call handler
        
        if(proc->killed)
            exit();
    }
    // Handle other types of traps (page faults, interrupts, etc.)
}
```

**Key Operations:**
1. **Trap Type Check**: Determines if this is a system call (`T_SYSCALL`)
2. **Process Kill Check**: Ensures the process isn't marked for termination
3. **Trap Frame Storage**: Saves the trap frame in the current process structure
4. **System Call Handler**: Calls the generic `syscall()` function
5. **Final Kill Check**: Checks again if the process should be terminated

### Process Structure Integration
The current process's trap frame is stored in the process structure for easy access by system call handlers and to maintain process context across system calls.

### The syscall() Function in syscall.c
This is where the actual system call routing happens:

```c
void syscall(void) {
    int num;
    
    // Extract system call number from a7 register
    num = p->trapframe->a7;
    
    // Validate system call number
    if(num >= 0 && num < NELEM(syscalls) && syscalls[num]) {
        // Call the appropriate system call handler
        p->trapframe->a0 = syscalls[num]();
    } else {
        // Invalid system call number
        printf("%d %s: unknown sys call %d\n", 
               p->pid, p->name, num);
        p->trapframe->a0 = -1;  // Return error
    }
}
```

**Critical Steps:**
1. **Extract System Call Number**: Gets the number from the `a7` register in the trap frame
2. **Validation**: Ensures the number is valid and within bounds
3. **Function Call**: Uses the system call table to call the appropriate handler
4. **Return Value**: Places the return value back in the `a0` register

### System Call Number Extraction
Remember the user-space code that put the system call number in `a7`? Here's where it's retrieved:

```c
num = p->trapframe->a7;
```

**The Journey of the System Call Number:**
1. User code: `li a7, SYS_exit`
2. Hardware saves `a7` in trap frame during trap
3. Kernel extracts: `num = p->trapframe->a7`
4. Kernel uses number to route to correct handler

### Return Value Mechanism
The return value follows the same path in reverse:

```c
p->trapframe->a0 = syscalls[num]();
```

**The Return Value Journey:**
1. System call handler returns a value
2. Kernel places value in trap frame: `p->trapframe->a0 = return_value`
3. Hardware restores `a0` from trap frame on return
4. User code receives return value in `a0`

### Error Handling
Invalid system call numbers result in error messages and return -1, demonstrating the kernel's defensive programming approach.

## System Call Table and Routing Mechanism

### The syscalls Table
The heart of system call routing is the `syscalls` table - an array of function pointers:

```c
// In syscall.c
static int (*syscalls[])(void) = {
[SYS_fork]    sys_fork,
[SYS_exit]    sys_exit,
[SYS_wait]    sys_wait,
[SYS_pipe]    sys_pipe,
[SYS_read]    sys_read,
[SYS_kill]    sys_kill,
[SYS_exec]    sys_exec,
[SYS_fstat]   sys_fstat,
[SYS_chdir]   sys_chdir,
[SYS_dup]     sys_dup,
[SYS_getpid]  sys_getpid,
[SYS_sbrk]    sys_sbrk,
[SYS_sleep]   sys_sleep,
[SYS_uptime]  sys_uptime,
[SYS_open]    sys_open,
[SYS_write]   sys_write,
[SYS_mknod]   sys_mknod,
[SYS_unlink]  sys_unlink,
[SYS_link]    sys_link,
[SYS_mkdir]   sys_mkdir,
[SYS_close]   sys_close,
};
```

**Key Features:**
- **Indexed Access**: System call number directly indexes into this array
- **Function Pointers**: Each entry points to the actual system call implementation
- **Sparse Array**: Uses designated initializers (e.g., `[SYS_exit] sys_exit`)
- **Consistent Interface**: All handlers have the same signature: `int (*)(void)`

### System Call Numbers
System call numbers are defined as constants:

```c
// In syscall.h
#define SYS_fork    1
#define SYS_exit    2
#define SYS_wait    3
#define SYS_pipe    4
#define SYS_read    5
#define SYS_kill    6
#define SYS_exec    7
// ... and so on
```

**The Routing Process:**
1. User code puts system call number in EAX (e.g., `movl $SYS_exit, %eax`)
2. Kernel extracts number from trap frame: `num = proc->tf->eax`
3. Kernel uses number to index into syscalls table: `syscalls[num]`
4. Kernel calls the function pointer: `syscalls[num]()`

### System Call Handler Implementation
Each system call has its own handler function (e.g., `sys_exit`, `sys_chdir`):

```c
// Example: sys_exit in sysproc.c
int sys_exit(void) {
    exit();
    return 0;  // Never reached
}

// Example: sys_chdir in sysfile.c  
int sys_chdir(void) {
    char *path;
    if(argstr(0, &path) < 0)
        return -1;
    // ... implementation to change directory
    return 0;
}
```

**Handler Characteristics:**
- **Consistent Signature**: All return `int` and take no parameters
- **Parameter Extraction**: Use helper functions like `argstr()`, `argint()` to get arguments
- **Error Handling**: Return -1 for errors, 0 or positive values for success
- **Location**: Organized by category (e.g., `sysproc.c`, `sysfile.c`)

### File Organization
- **sysproc.c**: Process-related system calls (`fork`, `exit`, `wait`, `getpid`, etc.)
- **sysfile.c**: File system-related system calls (`open`, `read`, `write`, `close`, etc.)
- **syscall.c**: Generic system call routing mechanism

### Function Pointer Benefits
The syscalls table uses function pointers for simplicity, extensibility, performance, and maintainability - no large switch statements needed.

### Adding New System Calls
To add a new system call to xv6:

1. **Define the number**: Add to `syscall.h` (e.g., `#define SYS_hello 22`)
2. **Add to table**: Add entry to syscalls table (e.g., `[SYS_hello] sys_hello`)
3. **Implement handler**: Create `sys_hello()` function in appropriate file
4. **Update user space**: Add declaration to `user.h` and implementation to `usys.S`

This systematic approach ensures that all system calls follow the same pattern and are properly integrated into the kernel.

## Return Flow Back to User Space

### The Return Journey
After the system call handler completes, the kernel must carefully unwind all the work it did to get into the kernel and return control to the user program. This process is essentially the reverse of the trap entry process.

### Step-by-Step Return Process

#### 1. System Call Handler Completion
```c
// In syscall.c
proc->tf->eax = syscalls[num]();  // Return value placed in EAX
```
- The system call handler returns its result
- The return value is immediately placed in the EAX field of the trap frame
- This ensures the user program will receive the return value in the EAX register

#### 2. Return from syscall() Function
```c
// In trap.c
void trap(struct trapframe *tf) {
    if(tf->trapno == T_SYSCALL) {
        // ... process checking ...
        syscall();      // This function returns here
        // ... more checking ...
    }
}
```
- Control returns to the `trap()` function in `trap.c`
- Additional process termination checks may be performed
- The trap frame contains all the state needed to return to user space

#### 3. Return from trap() Function
- The `trap()` function returns to the assembly code in `trapasm.s`
- At this point, we're back in the `alltraps` routine
- The trap frame is still on the kernel stack with all saved state

#### 4. Assembly Code Return Sequence
The return sequence in `trapasm.s` reverses the entry process:

```assembly
# Return sequence (simplified)
# Restore all registers from trap frame
ld ra, 40(a0)
ld sp, 48(a0)
# ... restore other registers ...
ld a0, 112(a0)     # Restore a0 last (contains return value)
sret               # Supervisor return - the magic instruction
```

**Key Operations:**
- **Register Restoration**: Restore all general-purpose registers from the trap frame
- **Stack Restoration**: Restore user stack pointer
- **Supervisor Return**: The `sret` instruction performs the final transition

#### 5. The SRET Instruction
The `sret` (Supervisor Return) instruction is the counterpart to the `ecall` instruction:

**What SRET Does:**
- Restores the program counter from `sepc` CSR
- Restores the previous privilege mode from `sstatus` CSR
- Switches CPU back to user mode
- Continues execution where the `ecall` occurred

**Critical Point:**
- There are two special instructions for kernel transitions:
  - `ecall` (environment call): Gets into supervisor mode
  - `sret` (supervisor return): Gets out of supervisor mode
- Both are hardware-supported and perform privilege mode transitions

### Stack Transitions During Return

#### Before Return (In Kernel):
- Running on per-process kernel stack
- Trap frame contains saved user state
- CPU in kernel mode

#### After Return (In User Space):
- Running on user stack
- All registers restored to pre-trap values
- CPU in user mode
- `a0` register contains system call return value

### The Complete Round Trip
The system call mechanism forms a complete round trip:

**User → Kernel:**
1. User code: `li a7, SYS_exit; ecall`
2. Hardware switches to supervisor mode and kernel stack
3. Assembly code builds complete trap frame
4. C code handles system call and places return value in trap frame

**Kernel → User:**
1. C code returns from system call handler
2. Assembly code restores state from trap frame
3. `sret` instruction switches back to user mode and user stack
4. User code continues with return value in `a0`

### Error Conditions and Edge Cases

#### Process Termination
If the process is marked for termination during the system call:
- The process will call `exit()` instead of returning normally
- The process will never return to user space
- The scheduler will run a different process

#### Invalid System Calls
If an invalid system call number is provided:
- Error message is printed to console
- Return value is set to -1 in the trap frame
- Normal return process continues
- User program receives -1 as return value

### Performance Considerations
System calls involve significant overhead: mode switches, stack switches, register saving/restoration, and multiple kernel function calls. This is why system calls are expensive compared to regular function calls.

### Summary
The system call mechanism coordinates user-space libraries, assembly code, C kernel code, and hardware to provide secure, controlled access to system resources while maintaining security boundaries.
